# job-mail-scan — Docker & Run Instructions

This README explains how to run the project in Docker and where to place runtime configuration (secrets are not stored in the repository).

Summary
- Use the included `Dockerfile` and `docker-compose.yml` to build and run the application in a container.
- The runtime command is isolated in `docker/start-app.sh` (inside the image) and the host helper is `docker/run.sh`.
- Secrets (Google OAuth client ID/secret, credentials file) are intentionally not committed. Use environment variables, a git-ignored `.env`, or your CI/CD secret store.

Prerequisites
- Docker Desktop (or Docker Engine) with Compose support
- Bash for running the host helper scripts (WSL on Windows or Git Bash). You can also run `docker compose` commands directly from PowerShell.

Important files
- `Dockerfile` — multi-stage build that produces `/app/app.jar` and runs `docker/start-app.sh`.
- `docker/start-app.sh` — container entrypoint that executes the jar. This contains the actual java -jar command.
- `docker/run.sh` — host helper script for `docker compose` (usage: `./docker/run.sh up|down|logs`).
- `docker-compose.yml` — compose file that reads environment variables and mounts `./data/tokens` for Gmail tokens.
- `.env.example` — example env file. Copy to `.env` and fill in secrets (do NOT commit `.env`).
- `src/main/resources/credentials.json` — if you use Google client library credentials JSON, place it here (this file is gitignored).

Creating your `.env` (PowerShell)
1. Copy the example and edit it locally (Windows PowerShell):

```powershell
Copy-Item .env.example .env
notepad .env
```

2. Edit `.env` and set:
- `GOOGLE_CLIENT_ID` — your OAuth client ID
- `GOOGLE_CLIENT_SECRET` — your OAuth client secret
- `JVM_OPTS` — optional JVM options (example: `-Xms256m -Xmx512m`)

Do not commit `.env` to Git. The repository `.gitignore` already excludes it.

Run with docker-compose (recommended)
From the repository root, build and run in detached mode using the helper script:

```powershell
# Build and start
./docker/run.sh up

# Follow logs
./docker/run.sh logs

# Stop and remove containers
./docker/run.sh down
```

If you prefer to run `docker compose` directly in PowerShell:

```powershell
docker compose -f docker-compose.yml up --build -d
docker compose -f docker-compose.yml logs -f
docker compose -f docker-compose.yml down
```

Run the image directly (no compose)
1. Build the image:

```powershell
docker build -t job-mail-scan:latest .
```

2. Run the container and pass environment variables (PowerShell):

```powershell
docker run --rm -p 8080:8080 -e GOOGLE_CLIENT_ID="$env:GOOGLE_CLIENT_ID" -e GOOGLE_CLIENT_SECRET="$env:GOOGLE_CLIENT_SECRET" job-mail-scan:latest
```

Notes about secrets and credentials
- The project reads `GOOGLE_CLIENT_ID` and `GOOGLE_CLIENT_SECRET` from the environment (these are referenced in `src/main/resources/application.yaml` as placeholders). This is intentionally left out of source control.
- If you use a Google credentials JSON file for the Google client libraries, place it at `src/main/resources/credentials.json` (this path is gitignored). Inside the container the app expects `gmail.credentials-file: classpath:credentials.json` by default.
- Tokens generated by Google OAuth flows are stored under `./data/tokens` on the host (mapped to `/root/.credentials/job-mail-scan` in the container). This directory is gitignored.

GitLab CI / CI pipelines
- For CI/CD, store secrets in GitLab Project -> Settings -> CI/CD -> Variables. Add `GOOGLE_CLIENT_ID` and `GOOGLE_CLIENT_SECRET` there and mark them `Protected`/`Masked` as appropriate.
- The pipeline job environment will automatically expose these variables to `docker compose` / Docker build/run steps.

Advanced / production
- For production, prefer a secrets manager (HashiCorp Vault, AWS/Azure/GCP secret manager) and integrate via Spring Cloud Config or a startup script that injects secrets as environment variables.
- Consider using a non-root user in the container and a smaller JRE base image if you need a hardened production image.

Troubleshooting
- If the app fails to start due to missing OAuth config, confirm `GOOGLE_CLIENT_ID` and `GOOGLE_CLIENT_SECRET` are set in the environment the container receives (check `docker compose config` or container env via `docker exec env`).
- Make sure `src/main/resources/application.yaml` still contains placeholders (e.g. `${GOOGLE_CLIENT_ID:}`) rather than hard-coded secrets.
